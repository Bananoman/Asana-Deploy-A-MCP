Resumen ejecutivo

Su equipo nos presento un diagrama BPMN 2.0 con 9 departamentos, 6 fases, mas de 40 actividades y SLAs regulatorios de FDA, RASFF, FSA, CFIA, MHLW y FSANZ. Es un proceso complejo, y la pregunta de fondo era directa: puede Asana manejarlo?

La respuesta corta es que si, con matices. Asana no es un motor BPMN — no tiene gateways nativos, ni subprocesos con ciclo de vida autonomo, ni timers de countdown. Pero su Work Graph (secciones, custom fields, rules, proyectos vinculados) permite modelar el 85% del proceso de forma nativa. El 15% restante se cierra con cuatro soluciones concretas:

  • Script Actions de Asana para sincronizar subprocesos con el proceso padre, ejecutar escalamientos condicionales multi-nivel y generar notificaciones multi-jurisdiccion — todo dentro de la plataforma, sin servidores externos
  • Formula Custom Fields para metricas calculadas automaticamente (% de recuperacion, costos restantes, dias para cierre)
  • Middleware SAP HANA como proyecto de integracion separado para alimentar los campos fuente con datos de inventario y costos en tiempo real
  • AI Teammate "Script Actions Writer" para que su equipo genere nuevos scripts bajo demanda conforme evolucione el proceso

Esta propuesta describe con exactitud que cubre Asana, que requiere configuracion adicional y que necesita integracion externa. Sin promesas vagas.

El reto

Un recall Clase I por contaminante microbiologico es el escenario mas critico que enfrenta una empresa alimentaria. Hay riesgo real de dano grave o muerte, plazos legales de 24h para notificar a la FDA, y la necesidad de coordinar 9 departamentos en paralelo mientras se mantiene trazabilidad para auditorias posteriores y potencial litigio.

El proceso que su equipo nos compartio tiene tres complejidades especificas que van mas alla de la gestion de proyectos estandar:

Coordinacion paralela con sincronizacion obligatoria. Siete departamentos trabajan al mismo tiempo despues de que el Comite de Crisis activa el recall. No pueden avanzar al cierre hasta que todos los departamentos hayan completado sus tareas. Cualquier retraso en uno afecta al cierre del recall completo.

SLAs con consecuencias legales. No es lo mismo un "deadline interno" que un mandato de la FDA con consecuencias penales. El sistema necesita distinguir entre ambos tipos y escalar de manera diferenciada.

Subprocesos extensos. La logistica inversa en 6 paises tiene su propio flujo de aduanas, transportistas y certificadores de destruccion. Es un proyecto dentro del proyecto, con 20+ tareas propias y aprobaciones independientes.

Propuesta de mejora — cerrar los gaps

Asana resuelve el 85% del proceso de forma nativa. Los 4 gaps restantes se cierran con funcionalidades avanzadas de Asana y, donde sea necesario, integracion con su ecosistema SAP.

1. Script Actions para sincronizacion de subprocesos

El gap mas visible es la sincronizacion entre los 3 subprocesos y el proceso padre. Hoy eso requiere que alguien actualice manualmente la tarea ancla cuando el subproceso termina.

Script Actions de Asana (disponible en Enterprise/Enterprise+) es un entorno serverless Node.js que se ejecuta como parte de una regla. Cuando la ultima tarea de un subproceso se completa, el script identifica la tarea ancla en el proceso padre y la actualiza automaticamente — custom field, comentario y estado. Todo ocurre dentro de Asana, sin infraestructura externa.

El impacto es concreto: elimina aproximadamente 45 min/dia de actualizacion manual durante una crisis que puede durar 2-4 semanas.

2. Script Actions para escalamiento condicional multi-nivel

Cuando el porcentaje de recuperacion de producto no alcanza el 95%, se necesita escalar simultaneamente a tres departamentos: Comunicaciones (aviso publico adicional), Supply Chain (extension de logistica inversa) y Regulatorio (notificacion a reguladores). Las reglas nativas de Asana no pueden evaluar un campo numerico con threshold y crear multiples subtareas condicionalmente.

Un Script Action resuelve esto: lee el valor del campo, verifica que no existan escalamientos previos para evitar duplicados, y crea las tres subtareas en paralelo con sus respectivos assignees, due dates y prioridad critica.

3. Script Actions para logica multi-jurisdiccion

Cuando se activa un recall, se deben notificar N jurisdicciones segun los mercados afectados, cada una con su propio SLA regulatorio (FDA 24h, RASFF 48h, MHLW 72h, entre otros). El gateway inclusivo del BPMN dispara multiples caminos segun condiciones.

Un Script Action lee el campo multi-select de jurisdicciones y genera automaticamente una subtarea por cada regulador seleccionado, con el due date calculado segun el SLA correspondiente y los custom fields pre-configurados. Verifica duplicados antes de crear.

4. Formula Fields para metricas calculadas

Asana soporta Formula Custom Fields desde 2023. Esto permite calcular directamente en la plataforma:

  • % Recuperacion = (Unidades Recuperadas / Total Unidades) x 100
  • Costo Restante = Provision Total - Costo Acumulado
  • Dias para Cierre = Fecha Limite - Hoy

Los campos se actualizan automaticamente y se integran con los dashboards.

5. Middleware SAP HANA para datos de inventario

Las formulas de Asana calculan correctamente, pero los campos fuente (unidades devueltas, costos por categoria) vienen de SAP. Ni Make ni Zapier son adecuados para este nivel de integracion — su equipo opera con SAP HANA, no con herramientas intermedias.

Proponemos un middleware dedicado SAP → Asana que sincronice los datos relevantes via API. Es un proyecto de integracion separado que requiere trabajar en conjunto con su equipo SAP. Estimacion: 2-4 semanas, dependiendo de la complejidad de su landscape SAP.

6. AI Teammate "Script Actions Writer"

Para scripts futuros que su equipo necesite conforme el proceso evolucione, se configura un AI Teammate dedicado a generar codigo para Script Actions bajo demanda. Se le asigna una tarea con la descripcion del comportamiento deseado y genera el script listo para implementar. Esto le da autonomia a su equipo para extender la automatizacion sin depender de consultoria externa cada vez.

Narrativa para su equipo directivo: "Asana gestiona nativamente la gran mayoria del proceso de recall — modelamiento multi-departamental, SLAs con escalamiento, campos calculados para metricas, trazabilidad de decisiones y reporteria ejecutiva. Para la sincronizacion de subprocesos, el escalamiento condicional y las notificaciones multi-jurisdiccion, usamos Script Actions de Asana que mantienen todo dentro de la plataforma sin servidores externos. El unico punto que requiere integracion dedicada es la conexion con SAP HANA para datos de inventario en tiempo real."
