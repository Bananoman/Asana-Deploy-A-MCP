3. Prerequisitos

Antes de ejecutar cualquier flujo de prueba, cada uno de estos elementos debe estar configurado y activo. Si alguno falta, el flujo correspondiente fallara silenciosamente (sin error visible en la UI).

Reglas nativas de Asana

Configuradas manualmente en la UI de Asana, dentro de cada proyecto correspondiente.

  • R1 — SLA proximo a vencer: En proyecto Recall. Trigger: due date en 24h + Tipo SLA = Regulatorio. Accion: Estado SLA → Proximo + comentario.
  • R2 — SLA vencido regulatorio: En proyecto Recall. Trigger: due date vencido + Tipo SLA = Regulatorio. Accion: Estado SLA → Vencido + Prioridad Critica + subtarea de escalamiento.
  • R3 — SLA vencido interno: En proyecto Recall. Trigger: due date vencido + Tipo SLA = Interno. Accion: Estado SLA → Vencido + Prioridad Alta.
  • R4 — Entrada al War Room (Clase I): Cross-project. Trigger: Clasificacion = Clase I. Accion: agregar tarea al proyecto War Room.
  • R5 — Defaults en creacion: En proyecto Recall. Trigger: tarea creada. Accion: Prioridad = Alta, Fase = Deteccion, Estado SLA = En tiempo.
  • R6 — Approval aprobado: En proyecto Recall. Trigger: approval aprobado. Accion: mover a siguiente seccion + comentario de auditoria.
  • R7 — Entrada al War Room (SLA Vencido): Cross-project. Trigger: Estado SLA = Vencido. Accion: agregar tarea de alerta al War Room.

Script Actions

Instalados como Script Actions dentro de reglas del proyecto. Requieren Enterprise/Enterprise+.

  • Script Action 1 — Sincronizacion subproceso: En proyecto Logistica Inversa (1213986186148783). Trigger: tarea marcada completada. Archivo: script-1-sync-subprocess.js
  • Script Action 2 — Escalamiento triple: En proyecto Recall (1213986423006166). Trigger: custom field cambia (% Recuperacion). Archivo: script-2-triple-escalation.js
  • Script Action 3 — Multi-jurisdiccion: En proyecto Recall (1213986423006166). Trigger: tarea anadida a seccion Notificaciones. Archivo: script-3-multi-jurisdiction.js

AI Studio

  • F1 — Clasificador de incidencia: Trigger: formulario "Reporte de Contaminacion" completado. Accion: clasifica y asigna custom fields.
  • F2 — Generador de status: Trigger: tarea recurrente "Status Update" cada 4h. Accion: crea Status Update del proyecto.
  • F3 — Verificador de evidencia: Trigger: tarea movida a seccion Cierre. Accion: verifica adjuntos, marca campo Evidencia.

AI Teammates

  • Analista de Crisis: Configurado con acceso al proyecto Recall + subprocesos. Instrucciones de rol como sintetizador de status.
  • Script Actions Writer: Configurado para generar codigo de Script Actions bajo demanda.

Pasos de prueba

  1. Abrir el formulario "Reporte de Contaminacion" del proyecto Recall (1213986423006166).

6. Flujo 3: Notificaciones regulatorias multi-jurisdiccion

Escenario: La tarea de notificaciones se mueve a la seccion "3. Notificaciones" con las 7 jurisdicciones seleccionadas en el campo multi-enum. El Script Action 3 lee las jurisdicciones y genera una subtarea por cada regulador, con SLA especifico y due date calculado automaticamente.

Pasos de prueba

  1. Crear (o localizar) una tarea en el proyecto Recall con nombre "Notificaciones regulatorias — Lote L-2026-0847".
  2. Abrir la tarea y asignar el campo Jurisdiccion (1213986355488023) seleccionando las 7 opciones: FDA, RASFF, FSA, CFIA, MHLW, FSANZ, ARCSA.
  3. Mover la tarea a la seccion 3. Notificaciones (1213986355333127).
  4. Esperar 15-30 segundos a que el Script Action 3 se ejecute.
  5. Abrir la tarea y verificar que se crearon 7 subtareas.
  6. Verificar nombre, due date y custom fields de cada subtarea.

Comportamiento del Script Action 3

El script (script-3-multi-jurisdiction.js) ejecuta la siguiente logica:

  1. Lee la tarea con opt_fields: 'name,custom_fields,...multi_enum_values'
  2. Busca el campo Jurisdiccion (1213986355488023) y extrae las opciones seleccionadas
  3. Verifica que no existan subtareas previas con prefijo "Notificar " para evitar duplicados
  4. Para cada jurisdiccion, calcula el due date sumando las horas de SLA al momento actual:
    • FDA: 24h | RASFF: 48h | FSA: 48h | CFIA: 48h | MHLW: 72h | FSANZ: 72h | ARCSA: 24h
  5. Crea las subtareas en paralelo con Promise.all(), cada una con:
    • Tipo SLA = Regulatorio (1213986423408742)
    • Estado SLA = En tiempo (1213986386240818)
    • Departamento = Regulatorio (1213986186376769)
    • Prioridad = Critica (1213986378365788)
  6. Agrega un comentario resumen listando todas las jurisdicciones y sus SLAs
Que se dispara:
Script Action 3 Tarea anadida a seccion Notificaciones → lee multi-enum → crea N subtareas con SLA.
R13 Tarea en seccion Notificaciones → agrega como followers a Dir. Legal + Dir. Regulatorio.

8. Flujo 5: Escalamiento triple (% recuperacion < 95%)

Escenario: Durante el monitoreo, se actualiza el campo "% Recuperacion" a 42.5% en la tarea de monitoreo. Como esta por debajo del 95%, el Script Action 2 crea 3 subtareas de escalamiento simultaneo a Comunicaciones, Supply Chain y Regulatorio, cada una con prioridad critica.

Pasos de prueba

9. Flujo 6: Sincronizacion de subproceso (Logistica Inversa)

Escenario: El subproceso de Logistica Inversa (1213986186148783) tiene sus propias tareas internas. Pasos de prueba

  1. Abrir el proyecto Logistica Inversa (1213986186148783).
  2. Verificar que hay al menos 3-5 tareas de prueba creadas (ej. "Recoger producto — Ecuador", "Recoger producto — Colombia", "Certificacion de destruccion", etc.).
  3. Completar todas las tareas excepto una.
  4. En el proyecto Recall, verificar que la tarea ancla "[Logistica] Iniciar logistica inversa" no ha cambiado su campo Fase (el script solo se ejecuta cuando se completa la ultima).
  5. Completar la ultima tarea pendiente en Logistica Inversa.

14. Troubleshooting

Problemas con Script Actions

SintomaCausa probableSolucion
Script no se ejecuta al mover tarea La regla no tiene el trigger correcto o esta deshabilitada Ir a Project Settings → Rules → verificar que la regla esta activa y el trigger es el correcto (ej. "Task added to section")
Script se ejecuta pero no crea subtareas Ya existen subtareas con el prefijo esperado (proteccion anti-duplicados) Eliminar las subtareas existentes y volver a disparar. Revisar los logs del Script Action en la consola de reglas
Error "No se encontro tarea ancla" El keyword "[Logistica]" no esta en el nombre de la tarea ancla en el proyecto padre Verificar que la tarea en el proyecto Recall contiene exactamente "[Logistica]" en su nombre
Script timeout (no completa) Script Actions tienen timeout de ~20 segundos. Si el proyecto tiene muchas tareas, getTasksForProject puede tardar Reducir el numero de tareas en el proyecto de prueba a <50. Verificar que limit: 100 es suficiente
Error de permisos en el script El script intenta acceder a un proyecto donde el token no tiene acceso Verificar que el usuario del workspace tiene acceso a ambos proyectos (subproceso y padre)
Custom fields no se asignan Los GIDs de los custom fields no coinciden con los del workspace real Verificar GIDs en la seccion 13 de este documento. Los GIDs son especificos del workspace Favorita

Problemas con reglas nativas

SintomaCausa probableSolucion
Regla no se dispara La regla esta pausada o el trigger no coincide con la accion realizada Abrir Rules del proyecto, verificar estado (activa/pausada), revisar trigger y condiciones
Regla se dispara pero no ejecuta la accion La condicion no se cumple (ej. campo no tiene el valor esperado) Verificar que el custom field tiene el valor exacto que la regla espera. Los enum values son case-sensitive
Tarea no aparece en War Room R4 o R7 no esta configurada como cross-project rule Las reglas cross-project requieren que el proyecto destino (War Room) este accesible. Verificar permisos
Defaults no se asignan en tarea nueva R5 no esta activa o solo funciona para tareas creadas, no importadas/movidas Crear una tarea nueva (no mover una existente) para verificar que R5 funciona

Problemas con AI Studio

SintomaCausa probableSolucion
F1 no clasifica el formulario El flujo de AI Studio no esta vinculado al formulario correcto, o el plan no incluye AI Studio Verificar en AI Studio que el trigger es "Form submission" del formulario "Reporte de Contaminacion"

Como revisar logs de Script Actions

  1. Ir al proyecto donde esta la regla (ej. Recall para SA-2 y SA-3, Logistica Inversa para SA-1).
  2. Abrir Customize → Rules.
  3. Buscar la regla que contiene el Script Action.
  4. Hacer clic en la regla → ver Activity log / Run history.
  5. Cada ejecucion muestra los mensajes de log() del script, incluyendo contadores de tareas, valores de campos, y errores.
