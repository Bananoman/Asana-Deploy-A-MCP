Resumen del caso

Demo solicitada por Jorge Romero para una empresa alimentaria multinacional (Ecuador). El proceso: Product Recall Clase I por contaminante microbiologico, con 9 departamentos, 6 fases, SLAs regulatorios de FDA/RASFF/FSA/CFIA/MHLW/FSANZ. El cliente envio un BPMN 2.0 completo y tiene expectativas de BPM enterprise.

El roadmap tecnico ya esta listo. Este documento cubre lo que falta para llevar el entorno de demo a produccion.

10 pasos para ir a produccion

Esta lista esta pensada para imprimir y usar como checklist durante la implementacion. Cada paso tiene una dependencia clara y un entregable verificable.

  1. ☐Confirmar plan de Asana del cliente Verificar si el cliente tiene Enterprise o Enterprise+ (necesario para Script Actions y AI Teammates con instrucciones avanzadas). Si tiene Business, hay funcionalidad limitada en reglas y AI. Esto determina el techo de lo que podemos configurar.
  2. ☐Crear workspace o acceder al del cliente Solicitar acceso como admin al workspace del cliente. Si prefieren un workspace nuevo para la fase piloto, crearlo con el dominio correcto. Necesitamos permisos para crear proyectos, custom fields globales y reglas.
  3. ☐Levantar la estructura base Crear el proyecto principal "Recall Clase I" con 6 secciones. Crear los 3 subproyectos (Logistica Inversa, Causa Raiz, Contact Center). Crear el proyecto War Room. Configurar los 10 Custom Fields globales. Esto es mecanico y toma medio dia.
  4. ☐Cargar tareas y configurar dependencias Crear las 30+ tareas pre-definidas del template. Establecer dependencias entre fases (Fase 2 no arranca sin Approval de Fase 1). Crear el milestone de sincronizacion en Fase 6 con dependencias de todos los cierres departamentales. Vincular tareas ancla a subproyectos via multi-homing.
  5. ☐Implementar las 15 reglas Configurar R1-R15 en el orden del roadmap. Probar cada regla individualmente con una tarea de prueba antes de pasar a la siguiente. Las reglas R14 y R15 (cross-project) son las mas delicadas porque dependen de que el War Room exista y tenga las secciones correctas.
  6. ☐Configurar AI Studio y AI Teammate Crear los 3 flujos de AI Studio (F1-F3). Crear el AI Teammate "Analista de Crisis" con las instrucciones del roadmap. Probar F1 con un formulario de prueba para validar que la clasificacion funciona. El AI Teammate necesita 2-3 interacciones de calibracion.
  7. ☐Construir el formulario de intake Crear el formulario "Reporte de Contaminacion" con los 7 campos. Vincular al flujo F1 de AI Studio. Probar end-to-end: llenar formulario, tarea creada, AI clasifica, asigna a Dir. Calidad.
  8. ☐Armar el dashboard ejecutivo 6 widgets: semaforo de SLAs, SLAs regulatorios vencidos, progreso por departamento, tareas criticas pendientes, recuperacion por jurisdiccion, costos acumulados. El widget de recuperacion por jurisdiccion requiere una tarea por jurisdiccion con el campo % actualizado — documentar esto para el equipo del cliente.
  9. ☐Cargar mock data y simular recall completo Cargar datos realistas: Lote L-2026-0847, contaminante Salmonella, 6 jurisdicciones, costos estimados. Ejecutar el recall completo paso a paso: formulario, clasificacion, activacion del Comite, notificaciones, ejecucion paralela, escalamiento por < 95%, cierre. Documentar cualquier regla que no dispare correctamente.
  10. ☐Guardar como template y documentar Una vez validado, guardar el proyecto como Project Template. Escribir una guia de uso de 2 paginas para el equipo del cliente: como instanciar un recall, que campos llenar, como interpretar el dashboard, cuando escalar manualmente. El template es el entregable final — el cliente puede replicar el proceso para cada recall futuro sin depender de nosotros.

Estimacion de tiempo (3 escenarios)

El tiempo depende de la velocidad de respuesta del cliente y de si tienen Enterprise+ (para Script Actions).

EscenarioDuracionQue incluyeRequisitos
Rapido10 dias habilesEstructura + reglas + AI Studio + AI Teammate + dashboard + template + rehearsal. Sin Script Actions ni integracion SAP.Plan Business o superior. Acceso al workspace en dia 1. Cliente responde en < 24h.
Completo15 dias habilesTodo lo del escenario rapido + Script Actions para sincronizacion de subprocesos + calibracion extendida del AI Teammate.Plan Enterprise/Enterprise+. Acceso al workspace en dia 1. Sesiones de calibracion con el equipo de Calidad.
Con SAP4-8 semanasTodo lo del escenario completo + middleware SAP HANA a Asana para datos de inventario y costos en tiempo real.Enterprise+. Acceso a endpoints SAP. Consultor SAP del lado del cliente. Definicion de alcance de integracion.
Recomendacion: Empezar con el escenario rapido (10 dias). Dejar que el equipo del cliente use el sistema durante 1-2 semanas de prueba. Despues evaluar si necesitan Script Actions y la integracion SAP con datos concretos de uso.

Riesgos y como mitigarlos

RiesgoProbabilidadImpactoMitigacion
Cliente tiene plan Business (no Enterprise)MediaAlto — sin Script Actions ni AI Teammates avanzadosConfirmar plan en el paso 1. Si es Business, ajustar roadmap para funcionar sin Script Actions (sincronizacion manual de subprocesos).
Expectativas de BPM enterprise que Asana no cumpleAltaMedio — frustraccion del clienteEl roadmap ya documenta cada limitacion con honestidad. En la demo, mostrar la limitacion y el workaround en el mismo momento.
Equipo SAP no disponible para integracionAltaBajo en fase 1 — la implementacion base no depende de SAPSeparar claramente la implementacion base (Asana) del proyecto de integracion (SAP). Avanzar con Asana primero.
AI Teammate genera outputs incorrectosMediaMedio — informacion erronea al Comite de CrisisCheckpoint obligatorio: el AI Teammate genera subtareas, pero un humano revisa antes de distribuir. Nunca envio automatico directo al Comite.
Reglas que no disparan en el orden esperadoMediaBajo — se corrige en la fase de rehearsalEl paso 9 (simulacion) es especificamente para detectar y corregir estos casos. Reservar 2 dias completos para rehearsal.
